# Code-Quality Audit — pd-prep-for-pgdp backend (2026-05-16)

> **Status: FULLY RESOLVED** — All 42 findings addressed across 21 tasks (Tasks 1–17 shipped before 2026-05-16; Tasks 18–21 shipped 2026-05-16). Plan: `docs/superpowers/plans/2026-05-16-backend-quality-hardening.md`

Auditor: agent (research-only — no code changes).
Scope: every `.py` file under `src/pd_prep_for_pgdp/` (~12,800 LOC across 74 files).
Reference categories: silent exception swallowing, improper None/0 returns, getattr/hasattr abuse, duck-type magic, type-annotation gaps/lies, missing error propagation, other concerns.

---

## Executive summary

Five themes dominate, in rough order of severity:

1. **`core/ocr.py:_to_ocr_word` (lines 344-374) is a 30-line `getattr` cascade** that silently coerces every missing OCR field to zero. A `pd_book_tools.Word` whose bounding_box accidentally lost a `right` attribute becomes a `(0,0,0,0)` bbox with no log, no error, no test failure — silently producing a corrupt PGDP package. Worst offender in the repo.
2. **`core/illustrations.py` has the same `getattr(..., 0) or 0` pattern** (lines 53-67, 73-75) for region geometry — same silent-zero failure mode against the layout detector's `Region` shape.
3. **The dual-write path in `core/pipeline/page_stage_writer.py` uses `except BaseException` plus `contextlib.suppress(OSError)` extensively (lines 186, 188, 353, 356, 401, 424, 501, 503, 588, 686 etc.)**. The intent (preserve dual-write atomicity under `KeyboardInterrupt`) is sound but the rollback path itself silently swallows filesystem errors with no logging — a failed rollback during a crash leaves the on-disk state in an undefined intermediate without anyone knowing.
4. **`api/data/pages.py:get_page_text` (lines 401-407) and `update_page_text`/`delete_page_words` log + swallow words-blob decode errors** but return either `[]` or HTTP 500 with no structured error code. The `except Exception:` at line 405 catches everything including programmer errors (TypeError, KeyError) and converts them to "no words available" silently.
5. **`core/job_runner.py:_handle_project_run_dirty` (line 525) catches every page-level exception with bare `except Exception` and only logs at WARNING level, then marks the parent job as success if even one child succeeds.** A whole batch of dirty stage failures gets reported as a green job, with no per-page error surfaced to the API.

Beyond those, the code base shows pervasive use of `Any` typing (88 occurrences) on the registry / stage-impl boundary — partially justified by the dispatch design, but propagates `Any` into typed callers via `_call_impl` / `_load_parent_artifact`.

Total findings: **42** (5 Critical, 18 High, 19 Medium).

## Remediation plan

### Critical (fix before next release)

1. **#1 / #6 — replace getattr-cascades in `core/ocr.py` and `core/illustrations.py`**: produces silently-corrupt OCR output. Effort: **medium** (define one Protocol per consumer, add a `_word_from_pd_book_tools` adapter at the boundary, raise on missing fields, add a smoke test that exercises a pd-book-tools rename). The pd-book-tools dep is pinned (v0.9.0 per CLAUDE.md) so the contract is stable enough to type today.
2. **#22 — surface per-page failures in `_handle_project_run_dirty`**: today a "Run all dirty" pass with 50% failure rate looks green. Effort: **small** (aggregate child errors into `error_message`, set job status to `error` on any failure, write a unit test).
3. **#41 — stop leaking tracebacks in 500 responses**: real risk in managed mode. Effort: **small** (one-line gate on settings.mode, but needs a test for both branches).

Critical bucket estimate: **medium** total (~1-2 days for the OCR/illustrations refactor; <0.5 day for the other two).

### High (fix in next sprint)

Findings #2, #3, #5, #7, #9, #10, #12, #13, #15, #18, #20, #21, #25, #26, #30, #32, #34, #36, #38, #42.

These are mostly narrowing `except Exception`/`except BaseException` to the right exception type and surfacing the previously-swallowed information. They're individually small but collectively meaningful:

- **Narrow broad excepts** (#3, #5, #7, #9, #20, #25, #30, #32, #34): mechanical replacements. Each takes ~10 minutes; the test side is the work.
- **Surface error info to the API/UI** (#13, #21, #26, #36): each needs a new model field and a corresponding frontend change. Medium-effort because of the openapi-export cycle.
- **Page_stage_writer rollback** (#10, #12, #15): the dual-write contract is the load-bearing safety net; getting the rollback story consistent across the four `commit_stage_artifact*` paths is real work — refactor `_safe_rollback`, add tests that inject `OSError` on the cleanup path.
- **Path-traversal hardening** (#42): defensive; revisit when Windows support is reconsidered.

High bucket estimate: **medium-to-large** (~3-5 days end-to-end, including frontend type sync where the API contract shifts).

### Medium (fix in cleanup pass)

Findings #4, #8, #11, #14, #16, #17, #19, #23, #24, #27, #28, #29, #31, #33, #35, #37, #39, #40.

These don't change behaviour materially under the happy path; they make failures easier to diagnose and bugs harder to write.

- **`Any` and `getattr` cleanup in stage_runner.py** (#15, #16, #17, #19, #23, #24): largest medium-bucket item. The dispatch layer leans heavily on `Any` because it has to. The right boundary is a `StageImplProtocol` with overloaded variants per `output_type` — modest typing work, big readability win.
- **Error-code consistency** (#27, #29, #33): one-time pass to map exception classes → HTTP status codes consistently.
- **Logging / observability** (#28, #35, #37, #39): emit aggregate counters / circuit breakers for the silently-skipped paths.
- **Pydantic shape tightening** (#40): low-risk, mostly cosmetic until a wire client misbehaves.

Medium bucket estimate: **medium** (~2-3 days, easy to parallelise with the High bucket).

## Overall scoring

- **Critical**: 5 findings (#1, #6, #22, #41, plus #6's twin — really 3-4 root causes).
- **High**: 18 findings — dominantly broad-except patterns and silent-rollback paths.
- **Medium**: 19 findings — typing hygiene + observability.

The pattern across the audit is consistent: this codebase *catches errors well* but *propagates them poorly*. Almost every "high" item is the same shape — "we caught the right things, but the caller can't tell what happened." A single-day pass to add a `failure_kind` enum to job/stage state, plus narrowing the four worst `except Exception` blocks, would close ~80% of the High bucket.

The duck-typed boundaries with pd-book-tools (`Word`, `Region`, `RegionType`) are the only spots where silent-corruption is a real risk; everything else is "harder to debug than it should be" rather than "wrong on the wire." Those two files (`core/ocr.py:_to_ocr_word`, `core/illustrations.py:auto_detect_illustrations` + `_map_region_type`) deserve the first fix.
